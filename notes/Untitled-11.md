<h1>Python 3.12-3.15 bundling-relevant C API and internals delta</h1><h2>1. Embedding initialization API</h2><h3>PyConfig struct field additions</h3><p>The full <code>PyConfig</code> struct is defined in <code>Include/cpython/initconfig.h</code>. Fields present since 3.8 (PEP 587) that matter for bundlers - <code>isolated</code>, <code>use_environment</code>, <code>site_import</code>, <code>install_signal_handlers</code>, <code>use_hash_seed</code>, <code>hash_seed</code>, <code>write_bytecode</code>, <code>home</code>, <code>program_name</code>, <code>executable</code>, <code>base_executable</code>, <code>platlibdir</code>, <code>pycache_prefix</code>, <code>module_search_paths</code>, <code>module_search_paths_set</code>, <code>argv</code>, <code>exec_prefix</code>, <code>prefix</code>, <code>dev_mode</code> - remain unchanged across all four versions. <code>safe_path</code> and <code>stdlib_dir</code> were added in 3.11 and are stable through 3.15.</p><p>New fields added per version, relevant to embedders:</p><ul><li><p><strong>3.12</strong>: <code>int int_max_str_digits</code> (integer string conversion limit, <a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/issues/96512">gh-96512</a>), <code>int perf_profiling</code> (Linux perf support), <code>wchar_t *dump_refs_file</code>.</p></li><li><p><strong>3.13</strong>: <code>int cpu_count</code> (override <code>os.cpu_count()</code> via <code>-X cpu_count=N</code>). A bundled app in a container can report constrained CPUs.</p></li><li><p><strong>3.14</strong>: <code>int thread_inherit_context</code>, <code>int context_aware_warnings</code>. On Apple platforms only, <code>int use_system_logger</code> (routes stderr to Apple system log; <a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/issues/127592">gh-127592</a>, controversially backported to 3.13.2/3.12.9, <a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/issues/130940">gh-130940</a>). These fields change the struct layout, so the PyConfig ABI differs between 3.13 and 3.14.</p></li><li><p><strong>3.15</strong> (alpha): No new embedder-relevant fields. The significant change is removal of deprecated legacy APIs (see below).</p></li></ul><h3>PEP 741 PyInitConfig (added in 3.14)</h3><p><a target="_blank" rel="noopener noreferrer nofollow" href="https://peps.python.org/pep-0741/">PEP 741</a> introduces an opaque, string-keyed configuration API that accepts UTF-8 <code>const char *</code> instead of <code>wchar_t *</code>. Frozen modules system</h2><h3>struct _frozen layout</h3><p>Defined in <code>Include/cpython/import.h</code>.</p><p><strong>3.12</strong> (5 fields):</p><pre><code class="language-c">struct _frozen {
    const char *name;                 /* ASCII-encoded module name */
    const unsigned char *code;        /* marshalled bytecode, or NULL if deepfreezing */
    int size;                         /* byte count of code; negative = package (pre-3.11 convention) */
    bool is_package;                  /* true for packages (added 3.11) */
    PyObject *(*get_code)(void);      /* deepfreeze hook: returns static code object (added 3.11) */
};
</code></pre><p><strong>3.13, 3.14, 3.15</strong> (4 fields - deepfreeze removed, <a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/issues/108716">gh-108716</a>):</p><pre><code class="language-c">struct _frozen {
    const char *name;
    const unsigned char *code;
    int size;
    int is_package;                   /* changed from bool to int */
};
</code></pre><p>A bundler populating this struct for 3.12 must include the <code>get_code</code> field (set to <code>NULL</code> for traditional marshalled-bytecode entries). For 3.13 and later, the struct is simpler.</p><h3>PyImport_FrozenModules</h3><pre><code class="language-c">PyAPI_DATA(const struct _frozen *) PyImport_FrozenModules;
</code></pre><p>Global pointer to a NULL-terminated array of <code>struct _frozen</code>. Set this <strong>before</strong> <code>Py_Initialize()</code> to make your frozen modules available. The array must end with a sentinel <code>{0, 0, 0}</code>. Internally, CPython also maintains private lists (<code>_PyImport_FrozenBootstrap</code>, <code>_PyImport_FrozenStdlib</code>, <code>_PyImport_FrozenTest</code>) that are searched alongside the public pointer. Custom entries via <code>PyImport_FrozenModules</code> are searched <strong>in addition to</strong> (not instead of) the internal lists (since <a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/commit/074fa5750640a067d9894c69378a00ceecc3b948">commit 074fa57</a>, Python 3.11).</p><p><strong>No changes to this mechanism from 3.12 through 3.15.</strong> It remains the sole stable way to register frozen modules. There is no <code>PyInitConfig</code>-based equivalent - <code>PyInitConfig_AddModule()</code> handles only C extension modules, not frozen bytecode. A community proposal for <code>PyImport_AppendFrozenModules()</code> was posted on <a target="_blank" rel="noopener noreferrer nofollow" href="https://discuss.python.org/t/add-stable-apis-for-easy-appending-of-frozen-modules-when-embedding/71362">discuss.python.org</a> (November 2024) but has not been implemented.</p><h3>Default frozen modules</h3><p>Source: <code>Python/frozen.c</code>, generated by <code>Tools/build/freeze_modules.py</code>. The list is <strong>stable across 3.12-3.15</strong>:</p><p><strong>Bootstrap</strong> (always frozen, regardless of <code>use_frozen_modules</code>): <code>_frozen_importlib</code> (importlib._bootstrap), <code>_frozen_importlib_external</code> (importlib._bootstrap_external), <code>zipimport</code>.</p><p><strong>Stdlib</strong> (frozen when <code>use_frozen_modules=1</code>): <code>abc</code>, <code>codecs</code>, <code>io</code> (needed without site), <code>_collections_abc</code>, <code>_sitebuiltins</code>, <code>genericpath</code>, <code>ntpath</code>, <code>posixpath</code>, <code>os</code>, <code>site</code>, <code>stat</code> (needed with site), <code>importlib.util</code>, <code>importlib.machinery</code>, <code>runpy</code> (needed for <code>-m</code>).</p><p><strong>Not frozen</strong>: <code>encodings</code> (commented out due to build noise), <code>importlib.abc</code>. A bundler that needs these modules must provide them separately.</p><h3>Deepfreeze timeline</h3><ul><li><p><strong>3.11</strong>: Added. Generated C structs for code objects instead of marshalled bytes. Output in <code>Python/deepfreeze/</code>. The <code>get_code</code> field in <code>struct _frozen</code> pointed to a function returning the static code object.</p></li><li><p><strong>3.12</strong>: Still active and default for release builds. Contributed roughly 300KB savings by excluding marshalled fallback data.</p></li><li><p><strong>3.13</strong>: <strong>Removed</strong> (<a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/issues/108716">gh-108716</a>). Key PRs: <a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/pull/108722">#108722</a> (turned off), <a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/pull/110078">#110078</a> (removed build artifacts), <a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/pull/116919">#116919</a> (cleaned up infra), <a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/pull/117141">#117141</a> (removed <code>_PyStaticCode_Init/Fini</code>). Rationale: slow to build (25-75% of build time), slightly slower than regular freezing in practice, incompatible with other optimizations.</p></li><li><p><strong>3.14, 3.15</strong>: Not present. Regular marshalled freezing is the sole mechanism.</p></li></ul><p>For a bundler, deepfreeze is irrelevant on 3.13 and later. Use marshalled bytecode arrays via the standard <code>struct _frozen</code> mechanism.</p><h3>Programs/_freeze_module</h3><p>The <code>Programs/_freeze_module.c</code> tool takes a <code>.py</code> file, initializes a minimal Python interpreter, compiles the source with <code>Py_CompileStringExFlags()</code>, marshals the code object with <code>PyMarshal_WriteObjectToString()</code>, and writes the result as a C <code>unsigned char</code> array to a <code>.h</code> file. The output files live in <code>Python/frozen_modules/*.h</code> and are included by <code>Python/frozen.c</code>.</p><ul><li><p><strong>3.12-3.13</strong>: Uses <code>PyConfig</code> for its internal initialization.</p></li><li><p><strong>3.14-3.15</strong>: Switched to <code>PyInitConfig</code> API (PEP 741). Adds <code>assert(Py_MARSHAL_VERSION &gt;= 5)</code>.</p></li></ul><p>The legacy <code>Tools/freeze/</code> directory (old standalone freeze tool) is present in all versions but largely unmaintained.</p><h3>PyConfig.use_frozen_modules</h3><p>Controls whether the import system prefers frozen stdlib modules or loads from <code>.py</code> source. Values: <code>1</code> (on, default for installed Python), <code>0</code> (off), <code>-1</code> (auto: on if installed, off if running from source tree). The <code>-X frozen_modules=on|off</code> flag controls it from the command line. <strong>Bootstrap modules</strong> (<code>_frozen_importlib</code>, <code>_frozen_importlib_external</code>, <code>zipimport</code>) are <strong>always</strong> loaded frozen regardless of this setting. No behavioral changes across 3.12-3.15. Frozen modules lack <code>__file__</code> (show <code>&lt;frozen module_name&gt;</code> as origin), which breaks debugger breakpoints - this is why debugpy recommends <code>-Xfrozen_modules=off</code> for development.</p><hr><h2>3. Import system for custom loaders</h2><h3>sys.meta_path default contents (unchanged 3.12-3.15)</h3><pre><code class="language-python">[BuiltinImporter, FrozenImporter, PathFinder]
</code></pre><p>A bundler's custom finder should be inserted at index 0 to intercept imports before filesystem search. For a fully hermetic bundle, remove <code>PathFinder</code> from <code>sys.meta_path</code> after init.</p><h3>Key import system changes</h3><ul><li><p><strong>3.12</strong>: <code>find_loader()</code> and <code>find_module()</code> removed from <code>zipimport.zipimporter</code>. Use <code>find_spec()</code>.</p></li><li><p><strong>3.13</strong>: <code>load_module()</code> deprecated on all loaders (use <code>exec_module()</code>). <code>__cached__</code> and <code>__package__</code> deprecated when set without matching <code>__spec__</code> attributes. Import system <strong>not</strong> available. Achieved by setting <code>PyConfig._init_main = 0</code> (private field).</p><p><strong>Phase 2 (main)</strong>: <code>_frozen_importlib</code> bootstrapped from frozen bytecode, then <code>_frozen_importlib_external</code>, then <code>sys.meta_path</code> populated, then path configuration applied, then site packages (if enabled).</p><pre><code class="language-c">PyConfig config;

// Insert custom finders here if needed

// Import system fully operational
</code></pre><p>The <code>_init_main</code>/<code>_Py_InitializeMain</code> split remains private and provisional in all four versions. The bootstrap order is: <code>_frozen_importlib</code> (core import machinery, <code>ModuleSpec</code>, <code>BuiltinImporter</code>, <code>FrozenImporter</code>) then <code>_frozen_importlib_external</code> (filesystem machinery, <code>PathFinder</code>, <code>FileFinder</code>, <code>SourceFileLoader</code>, magic number, .pyc handling). Both load from frozen bytecode compiled into the binary, not from disk.</p><hr><h2>5. zipimport changes</h2><h3>Compression methods</h3><table style="min-width: 125px;"><colgroup><col Built-in extension module registration</h2><h3>PyImport_AppendInittab (all versions)</h3><pre><code class="language-c">int PyImport_AppendInittab(const char *name, PyObject *(*initfunc)(void));
</code></pre><p>Part of the <strong>stable ABI</strong> (<code>Include/import.h</code>). Must be called before <code>Py_Initialize()</code>. In <strong>3.14</strong>, a <code>Py_FatalError</code> guard was added that kills the process if called after init. Returns <code>0</code> on success, <code>-1</code> on memory error. Internally calls <code>PyImport_ExtendInittab()</code>. <code>initfunc</code> is typically <code>PyInit_&lt;name&gt;</code> for both single-phase and multi-phase init modules. This is the primary mechanism for a bundler to register statically linked C extensions.</p><p>Also available: <code>PyImport_ExtendInittab(struct _inittab *newtab)</code> to add an entire NULL-terminated array at once, and direct assignment to the global <code>PyImport_Inittab</code> pointer (CPython-specific, not stable ABI).</p><pre><code class="language-c">struct _inittab {
    const char *name;
    PyObject *(*initfunc)(void);
};
</code></pre><h3>PyInitConfig_AddModule (3.14+, PEP 741)</h3><pre><code class="language-c">int PyInitConfig_AddModule(PyInitConfig *config, const char *name,
                           PyObject *(*initfunc)(void));
</code></pre><p>The <code>PyInitConfig</code> equivalent of <code>PyImport_AppendInittab</code>. Must be called after <code>PyInitConfig_Create()</code> and before <code>Py_InitializeFromInitConfig()</code>. Uses per-config state instead of a global, making it safer for multi-init scenarios. Not in the stable ABI (CPython-specific). For bundlers targeting 3.14 and later, this is the preferred API. For 3.12-3.13 compatibility, use <code>PyImport_AppendInittab</code>.</p><h3>PEP 489 multi-phase init status</h3><p><a target="_blank" rel="noopener noreferrer nofollow" href="https://peps.python.org/pep-0489/">PEP 489</a> (Python 3.5+) splits extension module init into creation phase (<code>Py_mod_create</code> slot) and execution phase (<code>Py_mod_exec</code> slot). Multi-phase init modules return a <code>PyModuleDef *</code> with <code>m_slots</code> from <code>PyInit_&lt;name&gt;</code>, while legacy single-phase init modules call <code>PyModule_Create()</code> and return a fully populated <code>PyObject *</code>. Both types work with <code>PyImport_AppendInittab</code> and <code>PyInitConfig_AddModule</code>. The stdlib migration to multi-phase init is ongoing (<a target="_blank" rel="noopener noreferrer nofollow" href="https://bugs.python.org/issue31862">bpo-31862</a>). Single-phase init remains supported without deprecation warnings as of 3.15, but a formal deprecation has been discussed (<a target="_blank" rel="noopener noreferrer nofollow" href="https://discuss.python.org/t/proposal-officially-deprecate-support-for-legacy-single-phase-init-extension-modules/89262">discuss.python.org thread</a>). Most popular third-party C extensions are still single-phase.</p><h3>PEP 793 PyModExport (3.15)</h3><p><a target="_blank" rel="noopener noreferrer nofollow" href="https://peps.python.org/pep-0793/">PEP 793</a> (accepted October 2025, Python 3.15) introduces a new entry point for C extensions:</p><pre><code class="language-c">PyModuleDef_Slot *PyModExport_&lt;NAME&gt;(void);
</code></pre><p>Instead of returning a <code>PyObject *</code> or <code>PyModuleDef *</code>, this returns a flat array of <code>PyModuleDef_Slot</code> structures with no enclosing <code>PyModuleDef</code>. This eliminates the requirement for a statically allocated <code>PyObject</code>. New helper functions:</p><pre><code class="language-c">PyObject *PyModule_FromSlotsAndSpec(const PyModuleDef_Slot *slots, PyObject *spec);
int PyModule_Exec(PyObject *module);
</code></pre><p>New slot types: <code>Py_mod_name</code>, <code>Py_mod_doc</code>, <code>Py_mod_state_size</code>, <code>Py_mod_methods</code>, <code>Py_mod_state_traverse</code>, <code>Py_mod_state_clear</code>, <code>Py_mod_state_free</code>, <code>Py_mod_token</code>.</p><p><strong>Impact on bundler inittab registration</strong>: PEP 793 does <strong>not</strong> change the inittab mechanism. <code>PyImport_AppendInittab</code> and <code>PyInitConfig_AddModule</code> still require a <code>PyInit_&lt;name&gt;</code> function. The PEP soft-deprecates <code>PyInit_*</code> (no warnings, fully supported, but no new features). Extensions can define both <code>PyModExport_&lt;name&gt;</code> and <code>PyInit_&lt;name&gt;</code> for backward compatibility. Python 3.15+ looks for <code>PyModExport_</code> first when loading from a shared library. For static linking via inittab, the <code>PyInit_</code> function is still what gets registered. The PEP lists inittab updates as a "Possible Future Direction."</p><hr><h2>7. Static linking libpython</h2><h3>Build system changes</h3><p>On Unix/Linux, the default CPython build produces a static <code>libpython3.X.a</code> and links the <code>python</code> executable against it. <code>--enable-shared</code> adds a shared <code>libpython3.X.so</code>. <code>--without-static-libpython</code> (added 3.10, <a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/issues/87269">gh-87269</a>) skips the static library when building shared.</p><ul><li><p><strong>3.12</strong>: Removed last vestiges of <code>setup.py</code>-based extension building. <code>Modules/Setup</code> is the sole mechanism for configuring built-in vs. shared extensions.</p></li><li><p><strong>3.13</strong>: Autoconf 2.71 required. Site initialization and path configuration</h2><h3>PyConfig.site_import</h3><p>Setting <code>site_import=0</code> (equivalent to <code>-S</code>) prevents import of the <code>site</code> module. What <code>site.py</code> normally does that a bundler wants to skip: appends <code>site-packages</code> directories to <code>sys.path</code>, processes <code>.pth</code> files, imports <code>sitecustomize.py</code> and <code>usercustomize.py</code>, adds <code>~/.local/lib/pythonX.Y/site-packages</code>, sets up <code>help()</code>/<code>quit()</code>/<code>exit()</code> builtins.</p><p><strong>3.14 change</strong>: <code>site.py</code> is no longer responsible for updating <code>sys.prefix</code> and <code>sys.exec_prefix</code> in virtual environments - this is now handled during path initialization by <code>getpath.py</code>. This means <code>sys.prefix</code> is correctly set even with <code>-S</code>, fixing a long-standing annoyance for embedders.</p><h3>PyConfig.safe_path (added 3.11)</h3><p>Corresponds to the <code>-P</code> flag (<a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/issues/57684">gh-57684</a>). When <code>1</code>, prevents prepending the script directory or current working directory to <code>sys.path</code>. Defense against code injection via CWD manipulation. <code>PyConfig_InitIsolatedConfig()</code> does <strong>not</strong> set <code>safe_path=1</code> by default; you must set it explicitly. The command-line <code>-I</code> flag (isolated) implies <code>-P</code>, but the C-level <code>config.isolated=1</code> does not automatically set <code>safe_path</code>.</p><h3>PyConfig.isolated</h3><p><code>PyConfig_InitIsolatedConfig()</code> sets: <code>isolated=1</code>, <code>use_environment=0</code>, <code>user_site_directory=0</code>, <code>site_import=0</code>, <code>install_signal_handlers=0</code>, <code>configure_locale=0</code>, <code>parse_argv=0</code>. Key effect: all <code>PYTHON*</code> environment variables are ignored. Subtlety for embedders (<a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/issues/101471">gh-101471</a>): even if you explicitly set <code>use_environment=1</code> after setting <code>isolated=1</code>, <code>getpath.py</code> may still ignore <code>pythonpath_env</code> because isolated mode causes <code>use_environment</code> to be reset during configuration resolution.</p><h3>PyConfig.module_search_paths_set and module_search_paths</h3><p>If <code>module_search_paths_set=1</code>, Python <strong>completely skips</strong> the path calculation algorithm and uses your list as <code>sys.path</code>. If <code>module_search_paths_set=0</code> (default), <code>Py_InitializeFromConfig()</code> overwrites whatever you put in <code>module_search_paths</code>. Since 3.11, <code>PyConfig_Read()</code> no longer pre-calculates path configuration fields, so you cannot append to a partially calculated list. No changes to this mechanism in 3.12-3.15.</p><h3>Path calculation algorithm (getpath.py)</h3><p>Since 3.11, path calculation is implemented in <code>Modules/getpath.py</code> (replacing the old per-platform C code in <code>getpath.c</code>/<code>getpathp.c</code>). The C wrapper <code>Modules/getpath.c</code> compiles and executes <code>getpath.py</code> as frozen bytecode. The algorithm:</p><ol><li><p>Check for <code>pyvenv.cfg</code> alongside or above the executable.</p></li><li><p>Check for a <code>._pth</code> file adjacent to the runtime library (if found, it precisely defines <code>sys.path</code>, enables isolated mode, and skips all env vars).</p></li><li><p>Check for <code>pybuilddir.txt</code> (build directory detection).</p></li><li><p>If <code>home</code> is set, use it for prefix/exec_prefix.</p></li><li><p>Search for landmark files relative to executable: <code>lib/pythonX.Y/os.py</code> (stdlib), <code>{platlibdir}/pythonX.Y/lib-dynload</code> (platlib).</p></li></ol><p><strong>Files probed</strong>: <code>pyvenv.cfg</code>, <code>python{major}{minor}.zip</code> (always added to path if found), <code>lib/pythonX.Y/os.py</code>, <code>{platlibdir}/pythonX.Y/lib-dynload</code>, <code>pybuilddir.txt</code>, <code>{name}._pth</code>.</p><p>Version-specific changes: <strong>3.14</strong> moved venv prefix detection from <code>site.py</code> into <code>getpath.py</code>, so <code>sys.prefix</code>/<code>sys.exec_prefix</code> are correct even with <code>site_import=0</code>.</p><p>Setting <strong>all</strong> path output fields explicitly (<code>prefix</code>, <code>exec_prefix</code>, <code>base_prefix</code>, <code>base_exec_prefix</code>, <code>stdlib_dir</code>, <code>executable</code>, <code>base_executable</code>, plus <code>module_search_paths</code> with <code>module_search_paths_set=1</code>) completely bypasses <code>getpath.py</code>. If even one is unset, the algorithm runs.</p><h3>PyConfig.platlibdir</h3><p>Platform-specific library directory name. Default <code>"lib"</code> on most systems, <code>"lib64"</code> on Fedora/openSUSE 64-bit, <code>"DLLs"</code> on Windows (since 3.11). Added in 3.9. Used by <code>getpath.py</code> to construct <code>{platlibdir}/pythonX.Y/lib-dynload</code>. No changes in 3.12-3.15. Set this to match your bundle's layout.</p><h3>PyConfig.pathconfig_warnings</h3><p>When <code>0</code>, suppresses <code>"Could not find platform independent libraries &lt;prefix&gt;"</code> and similar warnings during path configuration. Unix only. Default <code>1</code>. Set to <code>0</code> in a bundler to avoid confusing messages when expected paths do not exist.</p><h3>PYTHONHOME and PYTHONPATH in isolated mode</h3><p>Both are <strong>ignored</strong> when <code>use_environment=0</code>. Even explicitly setting <code>config.pythonpath_env</code> may be ignored because <code>getpath.py</code> guards usage with a <code>use_environment</code> check (<a target="_blank" rel="noopener noreferrer nofollow" href="https://github.com/python/cpython/issues/101471">gh-101471</a>). Workaround: populate <code>module_search_paths</code> directly with <code>module_search_paths_set=1</code>.</p><h3>Py_GetPrefix, Py_GetPath, etc. removal</h3><p><code>Py_GetPath()</code>, <code>Py_GetPrefix()</code>, <code>Py_GetExecPrefix()</code>, <code>Py_GetProgramFullPath()</code>, <code>Py_GetProgramName()</code>, <code>Py_GetPythonHome()</code> were deprecated in <strong>3.13</strong> and removed from public headers in <strong>3.15</strong> (kept in the binary for stable ABI compatibility). Replacements: <code>PyConfig_Get("module_search_paths")</code>, <code>PyConfig_Get("prefix")</code>, <code>PyConfig_Get("exec_prefix")</code>, <code>PyConfig_Get("executable")</code>, <code>PyConfig_Get("home")</code>, respectively.</p>
